systemd Production Unit

The production systemd unit (systemd/ch-api.service) is designed for secure, reliable operation on bare-metal Linux hosts with KVM support.

Key Configuration Decisions

  • Type=exec — Waits for the main process to start successfully before reporting active.
  • Restart=on-failure — Only restarts on unclean exits; avoids infinite loops on persistent config errors.
  • StartLimitBurst=3 + StartLimitIntervalSec=60 — Prevents restart storms.
  • Resource limits: LimitNOFILE=65536, MemoryMax=512M, TasksMax=512.
  • Security hardening: ProtectSystem=strict, ProtectHome=true, ReadWritePaths=/var/lib/ch-api, NoNewPrivileges=true, ProtectKernelTunables=true.
systemd/ch-api.service systemd unit file 
[Unit]
Description=Cloud Hypervisor API Service
Documentation=https://github.com/org/ch-api/docs/systemd.md
After=network.target

[Service]
Type=exec
User=ch-api
Group=ch-api
WorkingDirectory=/var/lib/ch-api
ExecStart=/usr/local/bin/ch-api
Restart=on-failure
RestartSec=5
StartLimitBurst=3
StartLimitIntervalSec=60

# Resource limits
LimitNOFILE=65536
MemoryMax=512M
TasksMax=512

# Security hardening
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/ch-api
NoNewPrivileges=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectControlGroups=true
RestrictRealtime=true
RestrictNamespaces=true
LockPersonality=true
MemoryDenyWriteExecute=true

# Logging
StandardOutput=journal
StandardError=journal
SyslogIdentifier=ch-api

[Install]
WantedBy=multi-user.target

Dockerfile

The Dockerfile uses a multi-stage build for a minimal production image:

  • Builder stage: golang:1.23-alpine with CGO_ENABLED=0 for static binary.
  • Runtime stage: gcr.io/distroless/static:nonroot — no shell, no package manager, minimal attack surface.
  • Non-root execution: UID/GID 65532 (distroless default nonroot user).
  • HEALTHCHECK: Uses /busybox/wget at /healthz endpoint.
Dockerfile Multi-stage container image 
# Build stage
FROM golang:1.23-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o ch-api ./cmd/ch-api

# Runtime stage
FROM gcr.io/distroless/static:nonroot
WORKDIR /app
COPY --from=builder /app/ch-api /app/ch-api
EXPOSE 8080 9090
USER nonroot:nonroot

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD ["/busybox/wget", "-qO-", "http://localhost:8080/healthz"]

ENTRYPOINT ["/app/ch-api"]

	@mkdir -p bin
	go build -ldflags="-w -s" -o bin/ch-api ./cmd/ch-api

	go test -race -coverprofile=coverage.out ./...

	golangci-lint run ./...

	go run ./cmd/ch-api

install: ## Install binary and systemd unit (requires sudo)
	install -Dm755 bin/ch-api /usr/local/bin/ch-api
	install -Dm644 systemd/ch-api.service /etc/systemd/system/ch-api.service
	useradd -r -s /usr/sbin/nologin ch-api || true
	mkdir -p /var/lib/ch-api /var/run/ch-api /etc/ch-api
	chown -R ch-api:ch-api /var/lib/ch-api /var/run/ch-api

uninstall: ## Uninstall binary and systemd unit (requires sudo)
	rm -f /usr/local/bin/ch-api /etc/systemd/system/ch-api.service
	userdel ch-api || true

Failure Scenarios

Scenario Symptom Key Remediation
2.1 Service fails to start Active: failed (Result: exit-code) Check port conflicts, config permissions, CH binary path, socket permissions, data directory
2.2 Service restarts in a loop Repeated restarts, same error on every boot systemctl stop ch-api, fix root cause, systemctl reset-failed ch-api
2.3 API returns 500 Endpoints return 500 with application/problem+json Check VMM reachability, SQLite locks, OOM kills, handler panics
2.4 High memory usage MemoryMax reached, OOM-killed Check goroutine leaks via pprof, restart service, increase MemoryMax temporarily
2.5 Authentication failures All calls return 401 Unauthorized Verify token secret, regenerate token, check RBAC roles, add Authorization header
2.6 Rate limiting 429 Too Many Requests with Retry-After Increase per-token limits in config or restart service to reset sliding windows
2.7 Disk space issues no space left on device Clean old event logs (>30 days), vacuum audit DB, move VM images to dedicated volume

GitHub Actions Release Workflow

The .github/workflows/release.yml automates the entire release pipeline on every v* tag push.

Workflow Steps

  1. Checkout — Fetch repository with full history for changelog generation.
  2. Setup Go — Use Go 1.23.x.
  3. Install Syft — For SBOM generation.
  4. Install Cosign — For keyless artifact signing.
  5. Login to registries — Docker Hub and GHCR using repository secrets.
  6. Run GoReleaser — Build, package, sign, push Docker images, publish GitHub Release.
.github/workflows/release.yml Automated release pipeline 
name: Release

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: write
  packages: write
  id-token: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.23.x'

      - name: Install Syft
        uses: anchore/sbom-action/download-syft@v0

      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Run GoReleaser
        uses: goreleaser/goreleaser-action@v6
        with:
          version: '~> v2'
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

See docs/releasing.md for the complete release process, verification steps, rollback procedure, and troubleshooting guide.
