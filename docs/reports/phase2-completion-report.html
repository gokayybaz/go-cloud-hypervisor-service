1. VMM Client Package

What was done

A production-ready Go client for the Cloud Hypervisor REST API was built using only the standard library (net/http, context, encoding/json). It supports dual transport (HTTP and Unix domain socket), automatic retries with exponential backoff and full jitter, structured error classification, and trace-ID-aware logging through the pkg/logging abstraction.

Files created or modified

  • pkg/vmm/client.go (334 lines) — Core client, request helpers, lifecycle methods
  • pkg/vmm/errors.go (147 lines) — Structured Error type with Op + Code classification
  • pkg/vmm/retry.go (53 lines) — Backoff delay computation and retry predicates
  • pkg/vmm/transport.go (96 lines) — HTTP/Unix transport configuration
  • pkg/vmm/types.go (121 lines) — VmConfig and nested CH API types
  • pkg/vmm/client_test.go (583 lines) — Unit tests for retry, errors, and transport

Key design decisions

Standard library only. No external HTTP client dependencies. The custom DialContext on http.Transport routes Unix socket requests transparently while still using net/http for URL building and header management.
Full jitter backoff. Prevents thundering herd during mass retries by randomising each delay in [0, exp) rather than using fixed exponential steps.
Op + Code dual classification. Every error carries both the operation name ("Create", "Boot") and the failure mode ("connection", "timeout", "api"). This eliminates string parsing when handling errors programmatically.

2. VM Lifecycle Methods

What was done

Seven high-level lifecycle operations were implemented on the vmm.Client: Create, Boot, Shutdown, Reboot, Pause, Resume, and Delete. Each operation is a thin wrapper around the corresponding CH REST endpoint that marshals/unmarshals JSON, handles retries, and logs every state transition.

Request/response structures

The VmConfig struct mirrors the Cloud Hypervisor REST API schema and includes:

  • CPUConfig — boot_vcpus, max_vcpus, optional topology
  • MemoryConfig — size, hotplug_method, hotplug_size, zones
  • KernelConfig — kernel image path
  • DiskConfig — path, readonly, direct I/O, queue counts
  • NetConfig — TAP, MAC, IP, mask, vhost-user
  • ConsoleConfig / SerialConfig — TTY, file, socket modes
  • PayloadConfig — firmware, kernel, cmdline, initramfs
  • BalloonConfig — size, deflate-on-OOM, free-page reporting
  • RngConfig — entropy source

VmInfo returns the running VM's current config, state, actual memory size, and CPU count.

Error handling approach

Every lifecycle method returns a *vmm.Error on failure. The error includes:

  • Op — which lifecycle method failed
  • Code — connection, timeout, api, not_found, retry_exhausted, invalid_request, unknown
  • Status — HTTP status code when applicable
  • Body — raw response body for debugging API errors
  • Cause — underlying error chain (supports errors.Is / errors.As)

Convenience helpers IsCreateFailed, IsBootFailed, etc. allow callers to branch on specific operation failures without parsing strings.

Trace-ID logging

Each lifecycle transition calls logTransition(ctx, op, state, ...), which derives a child logger via logger.WithContext(ctx). If the context carries a trace_id (injected by middleware.TraceIDMiddleware), every log line automatically includes it. Transition states logged are starting, succeeded, and failed.

3. Image Package

What was done

The pkg/image package manages VM disk images (format detection, validation, SHA-256 integrity), kernel command-line parsing with quote support and semantic validation, and cloud-init NoCloud datasource generation.

  • pkg/image/checksum.go (40 lines) — SHA-256 compute and verify
  • pkg/image/kernel.go (225 lines) — Kernel cmdline parser and validator
  • pkg/image/cloudinit.go (86 lines) — NoCloud datasource writer
  • pkg/image/image_test.go (411 lines) — Tests for all sub-packages

Disk image formats supported

FormatExtension(s)Constant
Raw.img, .rawFormatRaw
QCOW2.qcow2FormatQCOW2
VHDX.vhdxFormatVHDX
VMDK.vmdkFormatVMDK

Format is detected from the file extension. Image.Validate() resolves relative paths to absolute, verifies file existence, rejects directories, and optionally checks SHA-256.

Cloud-init integration

CloudInit.WriteConfigDrive(dir) creates the NoCloud datasource layout:

<dir>/user-data
<dir>/meta-data
<dir>/network-config   (optional)

Auto-generated meta-data falls back to instance-id and local-hostname when not explicitly provided. ToImage(dir) returns a read-only Image descriptor for attaching to vmm.DiskConfig.

Kernel cmdline validation rules

ParseCmdline splits on whitespace while respecting single and double quotes. Unbalanced quotes return an explicit error. Parameter keys must match ^[a-zA-Z_][a-zA-Z0-9_.\-]*$.

Semantic validation (Cmdline.Validate()):

  • root= must start with /dev/, UUID=, or LABEL=
  • ip= must be a valid IP address or CIDR

Mutable API: Get(key), Has(key), Set(key, value), AddFlag(key).

Image integrity verification

ComputeSHA256(path) streams the file through crypto/sha256. VerifySHA256(path, expected) compares the computed hash and returns a *ValidationError on mismatch, preventing accidental boot of corrupted images.

Edge cases handled

  • Unbalanced quotes in kernel cmdline → explicit error
  • Empty parameter keys → rejected
  • Invalid characters in keys → rejected by regex
  • Directory passed as image path → rejected with clear message
  • Missing image file → os.IsNotExist path with resolved absolute path

Validation rules

  • MAC: validated via net.ParseMAC; empty MAC triggers auto-generation of a locally-administered unicast address (02:xx:xx:xx:xx:xx)
  • IP: validated via net.ParseIP; optional field
  • QoS: ingress and egress rates must be non-negative
  • ID: auto-generated inside the write lock with collision retry (up to 10 attempts)

Audit log persistence

Every mutation is appended as NDJSON to the audit log file with:

{
  "timestamp": "2026-05-12T10:00:00Z",
  "operation": "AddInterface",
  "interface_id": "ethabcd",
  "details": { "mac": "02:...", "tap": "tap0", ... },
  "caller": { "file": "network.go", "line": 130, "func": "..." }
}

A separate sync.Mutex serialises log writes so audit I/O does not block read operations guarded by sync.RWMutex.

Problems encountered and resolutions

Concurrency collision in generateID. The initial time.Now().UnixNano()-based ID generator produced duplicate IDs under high concurrency (50 goroutines), causing AddInterface failures. Fixed by switching to crypto/rand and moving ID generation inside the write lock with a retry loop.

5. Storage Package

What was done

The pkg/storage package manages virtio-blk disks and snapshots for a single VM. It supports hot-plug (AddDisk), hot-unplug (RemoveDisk), snapshot creation (CreateSnapshot), and snapshot restore (RestoreSnapshot). Every operation is logged to a dedicated storage-ops.log with duration and outcome.

Snapshot storage strategy

Snapshots are implemented as full file copies using buffered io.Copy. This is the simplest portable approach: it works with any backing file format (raw, qcow2, etc.) and does not require qcow2 internal snapshot support or COW filesystems. The trade-off is that large disks require proportional time and space.

Log file format (storage-ops.log)

NDJSON lines with:

{
  "timestamp": "2026-05-12T10:00:00Z",
  "operation": "CreateSnapshot",
  "disk_id": "disk-abc",
  "duration_ms": 1523400,
  "outcome": "success",
  "message": "snap-a1b2c3d4",
  "caller": { "file": "storage.go", "line": 225, "func": "..." }
}

Limitations discovered

  • File-copy snapshots are not space-efficient. A 100 GB raw disk produces a 100 GB snapshot. Future phases may integrate qcow2 backing-file chains or filesystem snapshots (LVM, ZFS, btrfs) for COW efficiency.
  • No snapshot chaining or diffing. Each snapshot is an independent full copy; there is no parent/child relationship.
  • No atomic multi-disk snapshots. Snapshots are taken per-disk; a VM-wide consistent snapshot would require freezing I/O across all disks simultaneously.

6. Resources Package

What was done

The pkg/resources package manages vCPU and memory configuration for a single VM. It provides hot-resize operations with configurable limit and boundary validation, immediate in-memory updates, atomic persistent state storage, and structured resize event logging with before/after values and request source attribution.

Files created or modified

  • pkg/resources/resources.go (430 lines) — Manager, Config, Limits, operation log
  • pkg/resources/resources_test.go (652 lines) — 27 tests

Methods implemented

MethodDescription
ResizeCPU(count, source)Validate limits, update, persist atomically, log
ResizeMemory(sizeMB, source)Validate limits + alignment, update, persist, log
Config()Deep copy of current Config
SetLimits(limits) / Limits()Get/set validation boundaries
ReadLog()Read all resize log entries

Boundary and limit validation logic

Limits defines five boundaries:

FieldDefaultPurpose
MinCPUs1Minimum vCPU count
MaxCPUs1024Maximum vCPU count
MinMemoryMB64Minimum RAM
MaxMemoryMB1,048,576 (1 TiB)Maximum RAM
MemoryAlignMB4Alignment step (must divide evenly)

Validation is performed inside ResizeCPU and ResizeMemory under the write lock. Violations return a *resources.Error describing exactly which constraint failed (e.g. "cpu count 0 below minimum 1").

Clamping and rollback

There is no automatic clamping — out-of-bounds values are rejected with an error so callers are explicitly aware of the constraint. If the in-memory update succeeds but the atomic state persistence fails (disk full, permission denied), the manager rolls back the in-memory value to the previous state before returning the error. This prevents config drift.

Structured log fields

Every resize event appends NDJSON to resources-ops.log:

{
  "timestamp": "2026-05-12T10:00:00Z",
  "operation": "ResizeMemory",
  "source": "auto-scaler",
  "before": 256,
  "after": 512,
  "duration_ms": 1200,
  "outcome": "success",
  "message": "",
  "caller": { "file": "resources.go", "line": 212, "func": "..." }
}

The source field (e.g. "api", "cli", "auto-scaler") allows operators to attribute capacity changes to specific request origins. Empty source defaults to "unknown".

State persistence

State is stored as a single JSON file (resources.json) written atomically via temp-file + rename. On NewManager, existing state is loaded; otherwise defaults (1 CPU, 256 MB) are used and immediately persisted.
