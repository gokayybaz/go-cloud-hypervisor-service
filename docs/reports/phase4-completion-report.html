1. Audit Package

What was done

Created pkg/audit to persist every HTTP request to an operation_log table in SQLite. The database file rotates daily. A middleware captures the request body hash, extracts metadata from the request, and writes asynchronously with a 3-second background timeout.

Files created or modified

  • pkg/audit/audit.go — Auditor, SQLite schema, daily rotation
  • pkg/audit/middleware.go — HTTP middleware that hashes the body and persists metadata
  • pkg/audit/audit_test.go — Tests for DB creation, insert/query, rotation, body handling
  • pkg/api/router.go — Wired audit.Middleware into the middleware chain
  • cmd/ch-api/main.go — Initialized audit.New("data/audit")

SQLite operation_log schema

ColumnTypeNullableDescription
idINTEGERNOPrimary key, auto-increment
trace_idTEXTNORequest trace identifier
userTEXTYESActor from X-User-ID header
methodTEXTNOHTTP method (GET, POST, etc.)
pathTEXTNORequest path
body_hashTEXTYESSHA-256 hex of the first 1 MiB of body
duration_msINTEGERNOTotal request duration in milliseconds
status_codeINTEGERNOHTTP response status code
ipTEXTNOClient IP address
created_atDATETIMENODefaults to CURRENT_TIMESTAMP

Indexes: idx_trace_id on trace_id, idx_created_at on created_at.

Middleware chain position

PanicRecovery → TraceID → Audit → AccessLogger → Metrics → Timeout → Handler

Audit sits between TraceID and AccessLogger so it can read trace_id from context, and so AccessLogger does not include audit-insert latency.

Body hash computation

The middleware reads at most 1 MiB of the request body via io.LimitReader, computes a SHA-256 digest, encodes it as hex, then restores the original body so downstream handlers can still decode JSON. For empty bodies the hash is stored as an empty string.

Daily rotation strategy

  • Naming: data/audit-YYYY-MM-DD.db
  • Trigger: Rotation is checked on every Record call by comparing the current UTC date with the suffix of the open file.
  • Retention: No automatic deletion is implemented; operators are expected to archive or purge old files via standard log-rotation tooling (e.g. logrotate).
  • Performance tuning: PRAGMA journal_mode=WAL and PRAGMA synchronous=NORMAL are applied on every new file to keep writes fast without sacrificing durability.

2. Prometheus Metrics

What was done

Created pkg/metrics with a custom prometheus.Registry that registers five application metrics. A middleware records request count and latency per endpoint. An HTTP handler exposes the registry at /metrics.

Files created or modified

  • pkg/metrics/metrics.go — Registry, metric definitions, middleware, handler
  • pkg/metrics/metrics_test.go — Tests for all metrics and middleware
  • internal/handler/handler.go — Registered /metrics endpoint
  • cmd/ch-api/main.go — Created registry and passed to router/handlers/store
  • docs/metrics.md — Full reference with PromQL examples

Metrics reference

NameTypeLabelsDescription
chapi_http_requests_total Counter method, path, status Total number of HTTP requests received, labelled by method, path and status code.
chapi_http_request_duration_seconds Histogram method, path HTTP request latency distribution in seconds, labelled by method and path.
chapi_vms_active Gauge — Current number of VMs stored in memory.
chapi_api_errors_total Counter type Total number of API errors, labelled by semantic type.
chapi_vmm_request_duration_seconds Histogram op VMM socket round-trip latency in seconds, labelled by operation name.

Security & access restrictions

The /metrics endpoint is not authenticated. It is registered on the root router and is publicly accessible when the metrics registry is non-nil. There is no IP restriction or scrape-token gating. In a production deployment the endpoint should be protected by a reverse proxy (e.g. nginx allow-list) or by adding an authentication middleware.

Scrape recommendations

Documented in docs/metrics.md:

  • Scrape interval: 15–30 s
  • Alert on rate(chapi_api_errors_total{type="vmm"}[5m]) > 0.1
  • Dashboard panels for P95 latency, VM count, and VMM health are provided as PromQL snippets

3. Pprof Profiling

What was done

Created pkg/pprof as a lifecycle-managed HTTP server that exposes Go runtime profiling endpoints on a dedicated localhost:6060 address. It is disabled by default and must be enabled via a runtime configuration flag.

  • pkg/pprof/pprof_test.go — Tests for startup, shutdown, endpoint availability
  • internal/config/config.go — Added Profile bool field with default false
  • cmd/ch-api/main.go — Conditionally registers the pprof lifecycle component
  • docs/profiling.md — Complete profiling guide

/debug/pprof/symbolSymbol resolution

Listen address

localhost:6060 — hard-coded default. The server binds only to localhost, making it unreachable from remote hosts.

Production disable mechanism

Runtime flag, not a build tag. The profile configuration key (or CH_API_PROFILE=1 environment variable) controls whether the lifecycle component is registered. Default is false. A build tag was not used because runtime flags are more flexible for containerised deployments where operators may want to toggle profiling on a single pod without rebuilding the image.

Documented commands

From docs/profiling.md:

CPU profile

go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

Heap profile

curl -s http://localhost:6060/debug/pprof/heap > heap.prof
go tool pprof heap.prof

How make docs works

make docs
  1. Checks for $(GOPATH)/bin/pkgsite; installs golang.org/x/pkgsite/cmd/pkgsite@latest if missing

5. OpenAPI Spec

What was done

Integrated swaggo/swag to generate an OpenAPI (Swagger 2.0) specification from inline Go annotations. Added annotations to every handler method and mounted the Swagger UI at /api/docs. A CI step verifies that the committed spec matches the generated output.

Tool used

swaggo/swag — parses Go source files for structured comment annotations and emits a Swagger 2.0 spec. Note: swag outputs Swagger 2.0, not OpenAPI 3.1. The UI renders the 2.0 spec correctly and the format is fully supported by Swagger UI.

Output files

  • docs/docs.go — Go source embedding the JSON spec (imported by main.go)
  • docs/swagger.json — Raw OpenAPI JSON
  • docs/swagger.yaml — Raw OpenAPI YAML

Swagger UI serving

The UI is served at GET /api/docs/* via github.com/swaggo/http-swagger/v2:

router.Root().Get("/api/docs/*", httpSwagger.Handler(
    httpSwagger.URL("/api/docs/doc.json"),
))

Access URL: http://localhost:8080/api/docs/index.html

CI drift detection

The GitHub Actions workflow (.github/workflows/ci.yml) includes a step that:

  1. Installs swag CLI
  2. Runs swag init -g cmd/ch-api/main.go
  3. Runs git diff --exit-code docs/

If the committed spec diverges from the freshly generated output, the diff is non-empty and the CI job fails. This guarantees every PR either updates annotations and re-generates the spec, or leaves both untouched.

6. Event Log Package

What was done

Created pkg/eventlog to emit human-readable changelog-style entries for every VM operation. Each entry includes event type, VM ID, actor, and before/after state. Entries are written simultaneously to a daily-rotating log file and to syslog.

Files created or modified

  • pkg/eventlog/eventlog.go — Types, Writer interface, MultiWriter, Format()
  • pkg/eventlog/file.go — FileWriter with daily rotation
  • pkg/eventlog/syslog.go — SyslogWriter using log/syslog
  • pkg/eventlog/eventlog_test.go — Tests for format, rotation, multi-writer
  • internal/service/service.go — Added eventlog.Writer field
  • internal/service/vm.go — Emits events on CreateVM, DeleteVM, UpdateVM
  • internal/service/vm_lifecycle.go — Emits events on lifecycle transitions
  • cmd/ch-api/main.go — Initializes file + syslog writers, wraps in MultiWriter

Entry format

2026-05-13T21:30:00Z [boot] vm-abc by alice: status=created name="test-vm" cpus=1 mem=256MB disks=1 net=0 → status=running name="test-vm" cpus=1 mem=256MB disks=1 net=0

Sample entry (VM boot event)

2026-05-13T21:30:00Z [boot] vm-abc by alice: status=created → status=running

Dual output strategy

The MultiWriter broadcasts every entry to all configured backends:

  1. File — Appends to data/events-YYYY-MM-DD.log (daily rotating, same strategy as audit package)
  2. Syslog — Sends to local syslog daemon at LOG_INFO | LOG_DAEMON with tag ch-api

Individual backend errors are silently ignored so that a failing syslog connection never blocks a VM operation. If syslog is unavailable at startup, the server falls back to file-only and logs a warning.

Log file naming and rotation

  • Naming: data/events-YYYY-MM-DD.log
  • Rotation trigger: Checked on every Write call; if the current UTC date differs from the open file's date suffix, a new file is created
  • Retention: No automatic deletion; operators manage archival

Buffering and async strategy

None. Writes are synchronous. The MultiWriter.Write iterates backends in sequence and returns immediately. There is no buffered channel, no background goroutine, and no batching. This keeps the implementation simple and guarantees that an entry is persisted before the HTTP handler returns. The trade-off is that a slow syslog daemon adds latency to the request path; this is mitigated by the fact that errors are silently ignored and the write itself is a single UDP packet in typical syslog configurations.
