Key Design Decisions

  • cmd/ per binary — Each runnable binary gets its own subdirectory. This follows the standard Go project convention and keeps multiple binaries cleanly separated.
  • internal/ for private code — Domain logic (config, handler, service, store) lives under internal/ so it cannot be imported by external modules. The flow is handler → service → store.
  • pkg/ for public code — Reusable infrastructure (api, logging, middleware, preflight, lifecycle) lives under pkg/ and is safe for external consumption.
  • Go 1.22 ServeMux — Uses the enhanced http.ServeMux with method-and-path patterns (e.g. GET /healthz), avoiding a third-party router dependency.

Problems & Resolutions

Resolved: Go version mismatch go mod init initially wrote go 1.26.2 (matching the local toolchain). This was manually corrected to go 1.23.0 to match the requested 1.22+ baseline while staying current.
Resolved: Unused imports during initial build Early iterations of pkg/api/server.go and pkg/middleware/middleware.go had unused imports. These were caught by the Go compiler and removed before commit.

2. Preflight Package

Bare-Metal Hypervisor Readiness Checks

Completed
What was done: Implemented pkg/preflight to verify Cloud Hypervisor binary availability, kernel version, KVM access, and Linux process capabilities on the host.

Files

  • pkg/preflight/preflight.go — check implementations, report formatting
  • pkg/preflight/preflight_test.go — unit tests (3 tests)

Checks Implemented

CheckMethodExpected on Ubuntu 24.04
Binary availability exec.LookPath("cloud-hypervisor") + os.Stat mode check PASS if binary installed
Kernel version Read /proc/sys/kernel/osrelease, enforce >= 5.10 PASS (Ubuntu 24.04 ships 6.8)
KVM access Verify /dev/kvm exists, is a device node, and openable R/W PASS if user in kvm group
Capabilities (9 checks) Parse CapEff from /proc/self/status hex bitmask Individual PASS/FAIL per capability

Key Design Decisions

  • Zero external dependencies / no CGO — Capability bitmasks are parsed directly from /proc/self/status using only the standard library. A hard-coded map of 41 Linux capability constants is used instead of libcap bindings.
  • Structured report — Report aggregates Result values (name, ok, message). PrintFailed() emits only failures for concise operator feedback.
  • Dual-mode integration — PREFLIGHT=1 runs checks, prints the report, and exits (useful for systemd ExecStartPre). During normal startup, checks run silently; failures are logged as warnings but do not block the server unless strict_mode is enabled.

Current Host Results

Expected: All checks FAIL on current host (macOS/darwin) The development environment is macOS, not bare-metal Linux. Consequently:
  • /proc/sys/kernel/osrelease does not exist — kernel check FAILS
  • /dev/kvm does not exist — KVM check FAILS
  • /proc/self/status does not exist — capability checks FAIL
  • cloud-hypervisor is not installed — binary check FAILS
This is expected and correct. The package is designed for Ubuntu 24.04 bare-metal hosts and will pass there. The unit tests exercise the code paths without requiring a Linux host.

Problems & Resolutions

Resolved: Unused syscall import An initial draft imported syscall but never used it. The Go compiler rejected the build; the import was removed.

Key Design Decisions

  • Interface insulation — logging.Logger is a narrow interface. Callers never depend on zerolog directly, allowing the underlying library to be swapped without touching business logic.
  • No global logger — Every component receives its logger via dependency injection. Unit tests pass a NoOp implementation.
  • Caller skip tuning — zerolog.CallerSkipFrameCount = 3 in init() ensures caller points to the business code, not the wrapper internals.
  • JSON by default, console opt-in — Production environments use jq/Loki/Splunk. Pretty console output is strictly opt-in via LOG_FORMAT=console.

Trace-ID Propagation

middleware.TraceIDMiddleware injects a trace_id into every request context. It reads the X-Request-ID header (or generates a random 16-byte hex ID) and stores it via logging.WithTraceID. Handlers call logger.WithContext(r.Context()) so every log line carries the same trace_id.

Problems & Resolutions

Resolved: Switch from slog to zerolog The scaffold initially used Go 1.21's log/slog. It was replaced with zerolog for higher performance, a richer event API, and mature ecosystem integrations (Loki, Splunk). This added one external dependency, justified in ARCHITECTURE.md.

Hot-Reload

Implemented via Loader.Watch(onChange) which starts an fsnotify-based background watcher on the config file. When config.yaml is modified, Viper re-reads it and invokes the callback with the fresh Config. In main.go, the config-watcher lifecycle component starts this watcher; the callback currently logs the new log level. The watcher is stopped gracefully during shutdown.

Limitations

  • Environment variables are read once at startup and not re-evaluated on reload.
  • Structural changes (listen host/port) still require a process restart.
  • fsnotify may not work correctly on all network filesystems.

Key Design Decisions

  • Viper over custom parser — Viper provides a battle-tested layered config system (defaults to file to env) with built-in hot-reload, saving us from writing and maintaining a custom parser and file watcher.
  • mapstructure tags — All Config struct fields are tagged for mapstructure so Viper can unmarshal regardless of source (YAML, env, flag).
  • Missing file is non-fatal — If config.yaml is missing, Viper still has defaults and env vars. Only malformed files cause errors.

Problems & Resolutions

Resolved: Viper dependency not in go.mod after go get After running go get github.com/spf13/viper, the dependency was not persisted in go.mod because no source file imported it yet. Once internal/config/config.go was written with the import, go mod tidy correctly pulled in Viper and its transitive dependencies.
Resolved: fsnotify version mismatch The initially downloaded fsnotify was v1.9.0 (Viper's transitive dependency), but go mod tidy later resolved it to v1.10.1. This caused a brief build failure until go mod tidy was re-run to align all versions.

5. Lifecycle Package

Graceful Startup & Shutdown Orchestration

Completed
What was done: Implemented pkg/lifecycle with a Manager that handles ordered component startup, OS signal trapping (SIGINT/SIGTERM), and reverse-ordered shutdown with per-component timeout slices.

Files

  • pkg/lifecycle/lifecycle.go — Manager, Component interface, options
  • pkg/lifecycle/lifecycle_test.go — unit tests (4 tests)

Component Interface

type Component interface {
    Name() string
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
}

Signal Handling Strategy

The manager uses signal.NotifyContext to create a context that is cancelled when SIGINT or SIGTERM is received. The signals are configurable via WithSignals(). After receiving a signal, the manager logs the reason and proceeds to Phase 3 (shutdown).

Shutdown Order & Timeout

  • Reverse registration order — components stop in the opposite order they were registered. In main.go: config-watcher (reg #1) stops after http-server (reg #2).
  • Proportional timeout slices — The total shutdown timeout (default 30s, configurable via WithStopTimeout) is divided equally among components. Each component receives a sub-context with its slice. Minimum slice is 1s.
  • Timeout tolerance — If a component's Stop exceeds its slice, the manager logs a warning and continues with the next component rather than hanging forever.
  • Context error tolerance — context.DeadlineExceeded and context.Canceled errors from Stop are treated as timeouts, not failures.

Lifecycle Phases

  1. Startup — Components start in registration order. If any fails, already-started components are stopped in reverse order before returning the error.
  2. Wait — Block until OS signal or parent context cancellation.
  3. Shutdown — Stop all components in reverse order with timeout enforcement. Log every transition.

Components in main.go

OrderNameStartStop
1 config-watcher Starts Viper fsnotify watcher Calls Watcher.Stop()
2 http-server Starts http.Server.ListenAndServe() in goroutine Calls http.Server.Shutdown(ctx)

Key Design Decisions

  • Functional options pattern — NewManager(logger, lifecycle.WithStopTimeout(15s)) keeps the constructor clean and extensible.
  • Rollback on startup failure — If component N fails to start, components 1..N-1 are stopped before returning the error. This prevents leaked resources.
  • Logged transitions — Every start, stop, failure, and timeout is logged via the structured logger injected at manager creation.
  • Reusable abstraction — Replaces ad-hoc signal handling in main.go with a testable package that can be reused by other binaries or services.

Edge Cases Handled

  • No components registered — Manager logs a warning and waits for signal. Test: TestManagerNoComponents.
  • Startup failure mid-sequence — Already-started components are rolled back. Test: TestManagerStartupFailureRollback.
  • Stop timeout exceeded — Component respecting context returns ctx.Err(); manager treats this as a timeout warning, not a failure. Test: TestManagerStopTimeout.
  • Overall shutdown context expired — Manager logs remaining components and breaks out of the loop rather than waiting indefinitely.

Problems & Resolutions

Resolved: Startup rollback index calculation bug The initial implementation used indexOf() to determine which components to roll back, but this was fragile and unused. It was replaced with a simple started slice that tracks successfully started components, making rollback straightforward and correct.
Resolved: Stop timeout test took 5s The first version of TestManagerStopTimeout used time.Sleep(5 * time.Second) inside onStop, ignoring the context. This made the test take 5s. The component was rewritten to use a select on ctx.Done(), bringing the test runtime down to ~150ms.
