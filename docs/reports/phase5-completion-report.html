1. JWT Authentication Middleware

Completed

What was done

A JWT Bearer token authentication middleware was created in pkg/auth/auth.go. It validates tokens on every request to the /api/v1 sub-router, extracts claims into the request context, and returns structured 401 responses on failure.

JWT library

  • Library: github.com/golang-jwt/jwt/v5 (v5.3.1)
  • Only HS256 (HMAC-SHA256) is accepted.
  • HS256 was chosen for simplicity and because the service runs as a single node; asymmetric signing (RS256/ES256) would add key management complexity without a multi-service topology.

Token extraction

  • Header name: Authorization
  • Required prefix: Bearer (case-insensitive)
  • Malformed headers (missing prefix, wrong scheme) are rejected.

Claims validated

ClaimTypeRequiredDescription
substringRequiredSubject identifier; must be non-empty.
expNumericDateRecommendedExpiration time; tokens without exp are accepted but discouraged.
nbfNumericDateOptionalNot Before; standard validation applies.
iatNumericDateOptionalIssued At; standard validation applies.
issstringOptionalIssuer; validated when auth.issuer is configured.
audstring / []stringOptionalAudience; validated when auth.audience is configured.
roles[]stringOptionalRole strings for RBAC; injected into context.

Context injection

On successful validation the middleware injects two context values using typed context keys:

  • auth.SubjectFromContext(ctx) → string
  • auth.RolesFromContext(ctx) → []string

401 response body

{
  "type": "https://github.com/org/ch-api/errors/unauthorized",
  "title": "Unauthorized",
  "status": 401,
  "detail": "token expired",
  "instance": "/api/v1/vms"
}

The detail field varies to distinguish failure modes: "missing authorization header", "invalid authorization header format", "invalid token: ...", "invalid issuer", "invalid audience", "missing subject claim", "token expired".

Design decisions

  • Auth is mounted only on /api/v1 so that /healthz, /metrics, and /api/docs/* remain public.
  • Double expiry check: jwt.ParseWithClaims validates exp; a defensive second check ensures the token is not accepted even if the library is configured with leeway in the future.

Limitations

  • JWTs are not revocable without a blocklist. A compromised token remains valid until it expires.
  • Only symmetric signing (HS256) is supported. Key rotation requires a coordinated restart.

2. RBAC Authorization Layer

Completed

What was done

A role-based access control middleware was created in pkg/auth/rbac.go. It maps every protected endpoint to a minimum required role and enforces a three-tier hierarchy.

Roles defined

  • viewer (level 0) — read-only
  • operator (level 1) — viewer + lifecycle + resource changes
  • admin (level 2) — operator + create/delete + disk/network management

Higher roles implicitly inherit all permissions of lower roles. A user with ["viewer", "admin"] is treated as admin.

How the middleware reads roles

The RBAC middleware calls auth.RolesFromContext(r.Context()) to obtain the []string slice injected by the JWT middleware. Each string is parsed via auth.ParseRole; the highest numeric level is selected as the caller's effective role.

  "status": 403,
  "detail": "insufficient permissions: requires admin role",
  "instance": "/api/v1/vms"
}

Audit trail logging

The pkg/audit/middleware.go was updated to read the JWT sub claim from the context (auth.SubjectFromContext) before falling back to the X-User-ID header. Every authorization denial is:

  • Logged at WARN level with fields: reason, subject, user_roles, required_role, method, path.
  • Recorded in the SQLite audit trail with status code 403, trace_id, and the authenticated subject as the user field.

Design decisions

  • RBAC is opt-out via auth.rbac_enabled: false; default is true when auth is enabled.
  • Unlisted endpoints (e.g. GET /api/v1/status) are allowed if the token is valid, requiring no specific role.
  • Path matching uses segment-based wildcard rules rather than regex for performance and clarity.

3. Rate Limiting

Completed

What was done

A per-IP and per-token sliding-window rate limiter was created in pkg/ratelimit/ratelimit.go and mounted on the root router.

Algorithm

Sliding window counter.

For each key (IP or token subject) the limiter stores prevCount, currCount, and currWindow. On each request a weighted estimate is computed:

weight    = elapsed_time_in_window / window_size
estimated = prevCount * (1 - weight) + currCount

If estimated + 1 > limit, the request is rejected. This provides a smoother transition between fixed windows than a naive counter while remaining memory-efficient.

Data structure

In-memory map[string]*window protected by a sync.RWMutex. A background goroutine purges stale entries after each window expires.

Configuration

Limit TypeWindowMax RequestsApplies To
Per-IP1 minute100All requests
Per-token (viewer)1 minute100Tokens with viewer role
Per-token (operator)1 minute200Tokens with operator role
Per-token (admin)1 minute500Tokens with admin role

429 response

HTTP/1.1 429 Too Many Requests
Retry-After: 42
Content-Type: application/problem+json

{
  "type": "https://github.com/org/ch-api/errors/rate-limit-exceeded",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "rate limit exceeded for ip; retry after 42 seconds",
  "instance": "/api/v1/vms"
}

Logging

Every denial is logged at WARN with structured fields: type (ip/token), key, retry_after_sec, method, path, plus the trace_id from context.

Design decisions

  • Rate limiting is mounted on the root router so it also protects public endpoints (e.g. /healthz) from DoS.
  • IP extraction respects X-Forwarded-For and X-Real-Ip before falling back to RemoteAddr.
  • A limit set to 0 disables that specific limit while keeping others active.

Limitations

  • In-memory state is not shared across replicas. A distributed deployment would need Redis or similar.
  • IP-based limits can be bypassed by a client rotating IP addresses (e.g. botnets).

4. TLS Support

Completed

What was done

TLS configuration builder and ACME support were created in pkg/tls/tls.go. Both manual certificate mode and automatic Let's Encrypt mode are supported.

TLS Configuration Summary

  • Minimum TLS version: TLS 1.2
  • Certificate modes: Manual PEM files + ACME (Let's Encrypt)
  • Cipher suites: 6 AEAD suites with forward secrecy only
  • Excluded: CBC modes, RSA key exchange, TLS 1.0/1.1
  • Server preference: Enabled

Enabled cipher suites

TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305
TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305

CBC suites and RSA key exchange are excluded because they lack forward secrecy (RSA-KEX) or are vulnerable to padding oracle attacks (CBC). Only AEAD suites with ephemeral key exchange are permitted.

Certificate modes

ModeConfigNotes
Manualtls.cert_file + tls.key_fileServer loads PEM-encoded certificate and key at startup.
ACMEtls.acme_enabled: trueAuto-provisions certificates from Let's Encrypt. HTTP-01 challenge handler runs on port 80.

Non-TLS connections

When TLS is enabled the server calls http.Server.ListenAndServeTLS, which rejects unencrypted connections. There is no plaintext fallback.

Certificate reload

Manual certificates are loaded once at startup. A restart is required to reload them. ACME certificates are cached on disk in data/certs and renewed automatically by the autocert.Manager before expiry.

Design decisions

  • TLS hardening (min version, cipher suites) is applied unconditionally when TLS is enabled; there is no "permissive" mode.
  • The ACME HTTP-01 challenge server runs on port 80 in a separate goroutine and is not subject to the main middleware stack (it only serves ACME challenges).

5. Unix Socket Permission Check

Completed

What was done

A startup-time security check was added to pkg/preflight/preflight.go that verifies the Cloud Hypervisor VMM Unix domain socket is owned by and accessible only to the service user.

Exact assertions

#AssertionPass CriteriaFail Action
1Path is non-emptysocket_path != ""Fatal abort
2File exists and is a socketinfo.Mode() & os.ModeSocket != 0Fatal abort
3Owned by current userstat.Uid == os.Getuid()Fatal abort
4No group or other accessperm & 0o077 == 0Fatal abort

Syscalls / stdlib calls used

  • os.Stat(path) — stat the file
  • info.Sys().(*syscall.Stat_t) — extract raw Unix stat for UID
  • os.Getuid() — get effective UID of the current process
  • info.Mode().Perm() — extract permission bits

Logging

  • Pass: Logged at INFO level with message field containing the socket path, UID, and mode.
  • Fail: Logged at ERROR level with the failure reason, followed by immediate os.Exit(1).

Process behavior on failure

The process aborts immediately with exit code 1. There is no fallback, no retry, and no "strict mode" override. An insecure VMM socket is treated as a hard security boundary violation.

Design decisions

  • The check runs after general preflight checks but before any services (audit, HTTP server) are started. This prevents the API from accepting requests while the VMM socket is insecure.
  • The check is not conditional on preflight.enabled. Even if preflight checks are disabled, the socket check still runs because it is a security-critical invariant.
