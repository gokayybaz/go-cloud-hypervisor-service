1. HTTP Server Setup

What was done

A production HTTP server was built on the chi v5 router. All versioned REST endpoints live under /api/v1/ via a mounted sub-router. Non-versioned paths (e.g. /healthz) are registered on the root router.

Files created or modified

  • pkg/api/router.go (182 lines) — chi router, middleware stack, versioned sub-router
  • pkg/api/server.go (43 lines) — http.Server builder with ServerConfig
  • pkg/api/api_test.go (614 lines) — 19 tests for routing and middleware
  • internal/handler/handler.go (42 lines) — route registration wiring
  • cmd/ch-api/main.go — updated to use new router and server API

Server configuration

FieldDefaultDescription
Host"" (all interfaces)Bind address
Port"8080"Listen port
ReadTimeout30sMax duration for reading the entire request
WriteTimeout30sMax duration before timing out writes
IdleTimeout120sMax duration for next request on keep-alive

Key design decisions

chi over Go 1.22 ServeMux. chi provides URL parameters (/vms/{id}), middleware chaining, sub-router mounting, and method-based routing. The single third-party dependency is justified by the expressiveness it adds for RESTful API design.
TraceID before AccessLogger. The trace_id middleware runs before the access logger so that the access log can read trace_id from the request context. PanicRecovery is outermost so it catches panics from the goroutine spawned by Timeout.
Timeout runs the handler in a goroutine. This allows monitoring the deadline independently. If the handler panics in the goroutine, the panic is recovered and re-raised in the parent goroutine so PanicRecovery can handle it.

Problems encountered and resolutions

Panic not propagated through Timeout goroutine. The initial Timeout implementation swallowed panics from the handler goroutine. Fixed by adding a defer recover() inside the goroutine that stores the recovered value and re-panics it in the parent goroutine after <-done.

PATCH/vms/{id}/interfaces/{iface_id}NetworkHandler.PatchPartial update

MAC and IP/CIDR validation

  • MAC: validated via net.ParseMAC from the standard library
  • IP: first tried with net.ParseIP; if that fails, tried with net.ParseCIDR. Both must succeed for the value to be accepted.
  • Presence: at least one of tap, ip, or mac must be present on add.

PATCH semantics

Only provided fields are overwritten. Empty string values are ignored (not treated as "clear this field"). The response contains the current interface list.

7. Console WebSocket Endpoint

What was done

A WebSocket endpoint that streams simulated VM serial console output. Client disconnection is detected and cleaned up gracefully. Session start and end are logged with duration.

Files created or modified

  • internal/handler/console.go (108 lines) — WebSocket upgrade, streaming, cleanup
  • internal/handler/console_test.go (77 lines) — WebSocket dial and message tests
  • go.mod — Added github.com/gorilla/websocket

Endpoint

MethodPathHandlerProtocol
WS/api/v1/vms/{id}/consoleConsoleHandler.StreamWebSocket

Goroutine lifecycle

  1. VM existence is verified before WebSocket upgrade. If missing, a 404 Problem Detail is returned via HTTP.
  2. Connection is upgraded via websocket.Upgrader.
  3. A context.WithCancel is created from the request context.
  4. A goroutine runs conn.ReadMessage() in a loop. On any error (including close), it calls cancel() and returns.
  5. The main goroutine loops on a ticker (500ms), writing text messages. It also watches ctx.Done() and the done channel.
  6. On cleanup (context cancelled, done channel closed, or write error), the function jumps to the cleanup label, computes duration, logs session end, and returns. defer conn.Close() and defer cancel() handle resource cleanup.

Client disconnection

Disconnection is detected in the read pump. websocket.IsUnexpectedCloseError filters out normal closes (CloseGoingAway, CloseAbnormalClosure) from actual errors. Either way, the read pump calls cancel(), which signals the write pump to exit.
