1. VMM Client Unit Tests

What was done

Existing unit tests in pkg/vmm were enhanced and verified. The package now has 54 test functions exercising every public method of vmm.Client and all helper utilities.

Files

  • pkg/vmm/client_test.go — 54 test functions (enhanced/verified)
  • pkg/vmm/retry.go — retry logic helpers
  • pkg/vmm/transport.go — HTTP/Unix transport
  • pkg/vmm/errors.go — error types and helpers

Mock Unix socket server

Tests create a real Unix domain socket listener using net.Listen("unix", sockPath) and serve an http.ServeMux in a background goroutine. The client is configured with Transport: TransportUnixSock and Address: sockPath. This tests the custom DialContext in the transport layer without requiring a real CH process.

Test cases covered

  • Success paths: Ping, Version, Create, Boot, Shutdown, Reboot, Pause, Resume, Delete, Info
  • Timeout: TestClientTimeout — verifies context deadline causes ErrCodeTimeout
  • Retry: TestClientRetryOn5xx, TestClientRetryExhausted — tests 5xx retry logic and exhaustion fallback
  • Malformed responses: TestClientVersionError, TestClientCreateError, TestDoJSONUnmarshalError
  • Connection failures: TestClientNotFound, TestDoWithRetryConnectionError, TestWrapErrorDeadlineExceeded, TestWrapErrorCanceled, TestWrapErrorNetTimeout
  • Logging: TestClientLogsTransitions, TestClientLogsFailureWithTraceID, TestClientLogsRetryAttempts
  • Error helpers: IsCode, IsOp, IsNotFound, operation-specific helpers (IsBootFailed, etc.)
  • Retry policy: TestBackoffDelay, TestShouldRetryStatus, TestIsRetryableError

Race detector

go test -race ./pkg/vmm/...

All tests pass with the race detector. The client uses no shared mutable state; each request builds a fresh http.Request, making races impossible under normal use.

httptest setup

Binary detection and graceful skipping

func findCHBinary() string {
    if p := os.Getenv("CH_BINARY"); p != "" {
        if _, err := os.Stat(p); err == nil { return p }
    }
    if p, err := exec.LookPath("cloud-hypervisor"); err == nil { return p }
    return ""
}

If the binary is absent, every test calls t.Skipf("cloud-hypervisor binary not found..."). On macOS, tests also skip with t.Skip("Cloud Hypervisor requires Linux/KVM"). This makes go test ./... safe to run anywhere.

Local setup steps (from docs/e2e.md)

  1. Install or build cloud-hypervisor and add to $PATH
  2. Obtain a compiled kernel image (vmlinux or bzImage)
  3. Set environment variables: 
    export CH_KERNEL=/path/to/vmlinux
export CH_BINARY=/path/to/cloud-hypervisor  # optional if in $PATH
  4. Run tests: 
    go test ./e2e/... -v -timeout=120s

Timing and flakiness mitigation

E2E tests include fixed sleeps (200–500 ms) after state transitions to allow CH time to update internal state. The Shutdown test uses a polling loop with a 10-second deadline:

shutdownDeadline := time.Now().Add(10 * time.Second)
for time.Now().Before(shutdownDeadline) {
    info, err = client.Info(ctx)
    if err != nil || info.State == "Shutdown" { break }
    time.Sleep(pollInterval)
}

No flakiness was observed during local validation. The timeout-based polling makes the tests resilient to slower CI runners.

func FuzzCreateVMRequest(f *testing.F) {
    seed := []string{ /* 8 inputs */ }
    for _, s := range seed { f.Add([]byte(s)) }
    f.Fuzz(func(t *testing.T, data []byte) {

}

Initial corpus edge cases

Each target has 5–6 seed files representing:

  • Valid — well-formed JSON that passes validation
  • Empty — {} to test zero-value handling
  • Invalid fields — wrong types, missing required fields, out-of-range values
  • Non-JSON — plain text like "not-json" to ensure json.Unmarshal errors are handled
  • Boundary — zero, negative, or excessively large numeric values

Running fuzz tests locally

# Smoke test (5 seconds)
go test ./internal/handler/ -fuzz=FuzzCreateVMRequest -fuzztime=5s

# Full session (run until Ctrl-C)
go test ./internal/handler/ -fuzz=FuzzCreateVMRequest

# All targets briefly
for t in FuzzCreateVMRequest FuzzPatchCPURequest FuzzPatchMemoryRequest \
         FuzzAddInterfaceRequest FuzzPatchInterfaceRequest FuzzAddDiskRequest; do
    go test ./internal/handler/ -run='^$' -fuzz=$t -fuzztime=10s
done

Triage workflow (from docs/fuzzing.md)

  1. Discover: Fuzzer writes crash file to testdata/fuzz/<Target>/<hash>
  2. Reproduce: Run target against the crash file
  3. Minimize: go test -fuzzminimizetime=30s testdata/fuzz/.../<hash>
  4. Fix: Patch the handler/validation code
  5. Regression-test: Add minimized input to committed corpus, verify target passes

Known Issues & Limitations

  • pkg/preflight coverage is 24.6% because most checks require Linux-specific files (/proc/sys/kernel/osrelease, /dev/kvm, capabilities). The unit tests cover the parsing helpers but not the full check functions.
  • pkg/logging coverage is 49.2% because the zerolog-backed methods (Info, Error, Debug, Warn) are thin wrappers that are hard to unit-test meaningfully; coverage is achieved indirectly through integration tests.
  • E2E tests require a real kernel image (CH_KERNEL) for boot tests. Without it, those tests skip but the non-boot tests still run.
  • Cross-compilation for linux/amd64 uses CGO_ENABLED=0 because CGO is unavailable on macOS build hosts. This is acceptable because the project does not depend on C libraries.
