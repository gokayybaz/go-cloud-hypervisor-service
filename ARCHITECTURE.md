# Architecture Decisions

## Project Layout

```

├── go.work                # Go workspace file for multi-module support

│   └── ch-api/

├── internal/
```

### Decision: `cmd/` — Application Entrypoints

Each runnable binary lives in its own subdirectory under `cmd/`. The binary name matches the directory (`ch-api`). This follows the standard Go project convention and keeps multiple binaries cleanly separated if the project grows.

### Decision: `internal/` — Private Application Code

Packages under `internal/` cannot be imported by external modules (enforced by the Go toolchain). This is where all domain logic resides:

| Package   | Responsibility                                    |
|-----------|---------------------------------------------------|
| `config`  | Environment variable and configuration loading    |
| `handler` | HTTP handler registration, wiring routes to services |
| `service` | Core business logic, orchestrates store calls      |
| `store`   | Persistence abstraction, database interaction      |

The layered flow is: **handler → service → store**, keeping handlers thin and business logic testable in isolation.

### Decision: `pkg/` — Shared Public Packages

Packages under `pkg/` are safe for external consumption. They provide infrastructure that could be reused by other services:

| Package      | Responsibility                                    |
|--------------|---------------------------------------------------|
| `api`        | HTTP router wrapper and server helpers            |
| `image`      | VM disk images, kernel cmdline, cloud-init drives |
| `lifecycle`  | Graceful startup/shutdown orchestration           |
| `logging`    | Structured logging interface (zerolog)            |
| `metrics`    | Prometheus metrics registry and HTTP middleware   |
| `middleware` | HTTP middleware (trace ID injection, etc.)        |
| `pprof`      | Localhost-only Go profiling server (dev only)     |
| `eventlog`   | Human-readable changelog for VM operations        |
| `network`    | VM virtio-net interface management                |
| `preflight`  | Bare-metal hypervisor readiness checks            |
| `resources`  | VM vCPU and memory hot-resize management          |
| `storage`    | VM virtio-blk disk and snapshot management        |
| `vmm`        | Cloud Hypervisor REST API client                  |

### Decision: `docs/` — Documentation

End-user and developer-facing documentation including API specs and design rationale.

### Decision: chi Router (v5)

The `pkg/api` router uses [chi](https://github.com/go-chi/chi) v5 instead of Go 1.22's enhanced `http.ServeMux`. chi provides URL parameters (`/vms/{id}`), middleware chaining, sub-router mounting, and method-based routing with clean handler signatures. The single third-party dependency is justified by the expressiveness it adds for RESTful API design.

**Middleware stack (outermost → innermost):**
1. **PanicRecovery** — catches panics, logs stack trace via structured logger, returns 500
2. **TraceID** — reads `X-Request-ID` or generates one, stores in context, echoes on response
3. **Audit** — persists request metadata to SQLite operation_log (optional)
4. **AccessLogger** — structured access log with method, path, status, duration, bytes, trace_id
5. **Metrics** — Prometheus request count and latency histogram per endpoint
6. **Timeout** — per-request deadline (default 30s); writes 504 if exceeded

TraceID is placed *before* AccessLogger so the access log can read `trace_id` from the request context. PanicRecovery is outermost so it catches panics from the handler goroutine spawned by Timeout. Metrics sits just above Timeout so it captures handler latency without including timeout-monitoring overhead.

**Versioning:** A chi sub-router is mounted at `/api/v1`. Handlers register routes on `router.V1()`. Non-versioned routes (e.g. `/healthz`) are registered on `router.Root()`.

### Decision: Structured Logging via `zerolog`

The `pkg/logging` package wraps [`zerolog`](https://github.com/rs/zerolog) with a narrow `Logger` interface.  Every line automatically carries `timestamp`, `level`, `caller`, and `hostname`.  When a logger is derived from a request context via `WithContext`, a `trace_id` field is also injected.

- **JSON by default** — machine-readable for `jq`/Loki/Splunk.
- **Console mode** — set `LOG_FORMAT=console` for pretty-printed local development output.
- **Caller skip tuning** — `zerolog.CallerSkipFrameCount` is set to `3` so `caller` points to the business code, not the wrapper internals.
- **Interface insulation** — Callers depend on `logging.Logger`, not on zerolog directly, keeping the door open to swapping the underlying library.

### Decision: Go Workspace (`go.work`)

A `go.work` file is included from the start. This enables multi-module development should the project need separate modules (e.g., a proto module, a CLI tool module) without restructuring later. The single `use .` directive keeps the initial setup simple.

### Decision: Configuration via Viper (YAML + Env + Hot-Reload)

The `internal/config` package uses [Viper](https://github.com/spf13/viper) to provide a layered configuration system:

1. **Built-in defaults** — hard-coded in `setDefaults()`
2. **YAML config file** — `config.yaml` searched in `./`, `/etc/ch-api/`, `$HOME/.ch-api`
3. **Environment variables** — `CH_API_*` prefix with underscore nesting (e.g. `CH_API_SERVER_PORT`)

Environment variables have the highest precedence and override both defaults and file values. This aligns with the twelve-factor app methodology while also supporting file-based configuration for complex deployments.

**Hot-reload:** The `Loader.Watch()` method starts an `fsnotify`-based background watcher on the config file. When the file changes, Viper re-reads it and invokes a caller-provided callback with the fresh `Config`. This is useful for adjusting log levels or feature flags without restarting the process. Note that structural changes (e.g. listen port) still require a restart.

### Decision: Lifecycle Management (`pkg/lifecycle`)

The `pkg/lifecycle` package provides a `Manager` that orchestrates graceful startup and shutdown of application components.  Each component implements a simple `Component` interface (`Name`, `Start`, `Stop`).

**Key behaviors:**
- **Ordered startup** — components start in registration order; if any component fails to start, already-started components are stopped in reverse order before the error is returned.
- **Ordered shutdown** — on `SIGINT`/`SIGTERM` (or context cancellation), components stop in reverse registration order.  Each component receives a context with a proportional share of the total shutdown timeout.
- **Timeout-aware** — if a component's `Stop` exceeds its slice, the manager logs a warning and continues with the next component rather than hanging forever.
- **Context error tolerance** — `context.DeadlineExceeded` and `context.Canceled` errors from `Stop` are treated as timeouts, not failures.
- **Logged transitions** — every start, stop, failure, and timeout is logged via the structured logger.

**Components in `main.go`:**
1. `config-watcher` — starts the Viper file watcher for hot-reload
2. `http-server` — starts the HTTP listener; stops via `http.Server.Shutdown`
3. `pprof-server` (optional) — starts a localhost-only profiling server on `:6060` when `profile: true`

This replaces ad-hoc signal handling in `main.go` with a reusable, testable abstraction.

### Decision: Profiling (`pkg/pprof`)

A dedicated `pprof` package exposes Go runtime profiling endpoints on a separate
`localhost:6060` HTTP server.  It is **disabled by default** and must be
explicitly enabled via the `profile` config key or `CH_API_PROFILE=1`.

**Why a separate server?**
- Keeps profiling traffic off the public API port
- Binds only to `localhost` so remote exposure is impossible
- Can be enabled/disabled independently without affecting the API router

**Why not a build tag?**
A runtime flag is more flexible for containerised deployments where operators
may want to turn profiling on for a single pod without rebuilding the image.
The localhost-only binding provides defence-in-depth even if the flag is
accidentally left on.

### Decision: Trace ID Propagation

`middleware.TraceIDMiddleware` injects a `trace_id` into every request context.  It reads the `X-Request-ID` header (or generates a random ID) and stores it via `logging.WithTraceID`.  Handlers then call `logger.WithContext(r.Context())` so that every log line emitted during request processing carries the same `trace_id`.  This makes distributed debugging trivial.

### Decision: Preflight Checks (`pkg/preflight`)

A dedicated `preflight` package verifies that the host is ready to run Cloud Hypervisor workloads. This is critical for a bare-metal hypervisor API because failures are often environmental (missing KVM, wrong kernel, insufficient capabilities) rather than code bugs.

Checks performed:

1. **Binary availability** — `cloud-hypervisor` must exist in `$PATH` and be executable.
2. **Kernel version** — minimum Linux 5.10 (Ubuntu 24.04 ships 6.8, which satisfies this).
3. **KVM access** — `/dev/kvm` must exist, be a device node, and be openable read-write by the current process.
4. **Process capabilities** — effective capabilities are read from `/proc/self/status` and compared against a curated list required by Cloud Hypervisor (`CAP_SYS_ADMIN`, `CAP_NET_ADMIN`, `CAP_IPC_LOCK`, `CAP_SYS_NICE`, `CAP_SYS_RESOURCE`, `CAP_SYS_PTRACE`, `CAP_DAC_READ_SEARCH`, `CAP_SETUID`, `CAP_SETGID`).

The package uses **only the standard library** (no `libcap` bindings or CGO). Capability bitmasks are parsed from hexadecimal strings in `/proc/self/status` and compared against a hard-coded map of Linux capability constants.

**Integration:**
- Setting `PREFLIGHT=1` runs checks, prints a structured report, and exits (useful in init containers or systemd `ExecStartPre`).
- During normal startup, checks run silently; failures are logged as warnings but do not block the server. This allows operators to choose between fail-fast and best-effort modes.

**Structured report:**
- `Report` aggregates `Result` values (name, ok, message).
- `PrintFailed()` emits only failures for concise operator feedback.
- `Verify()` is a convenience wrapper that exits non-zero when any check fails.

### Decision: Cloud Hypervisor REST Client (`pkg/vmm`)

The `pkg/vmm` package provides a Go client for the Cloud Hypervisor REST API. It uses only the standard library (`net/http`, `context`, `encoding/json`) and supports both HTTP and Unix domain socket transports.

**Key features:**
- **Dual transport** — HTTP (`TransportHTTP`) or Unix socket (`TransportUnixSock`) via a custom `DialContext` on `http.Transport`
- **Exponential backoff with full jitter** — Retries on connection errors, timeouts, and HTTP 429/5xx. Full jitter prevents thundering herd during mass retries
- **Structured errors with operation context** — Every error is a `*vmm.Error` with `Op` (which operation failed, e.g. "Create"), `Code` (failure mode), HTTP status, raw response body, and cause chain. Operation-specific helpers (`IsCreateFailed`, `IsBootFailed`, etc.) allow precise error handling
- **Trace-ID-aware logging** — When a `logging.Logger` is configured, every lifecycle transition (start, success, failure, retry) is logged via `logger.WithContext(ctx)`, automatically injecting `trace_id` when present in the context
- **Context-aware** — All methods accept `context.Context`; cancellation during retry backoff is honoured immediately
- **Drain-and-close on retry** — Response bodies are consumed before retrying to prevent connection pool exhaustion

**API coverage:** Ping, Version, Create, Boot, Shutdown, Reboot, Pause, Resume, Delete, Info.

**Design decisions:**
- Standard library only — no external HTTP client dependency
- Request bodies are rebuilt on each retry (small JSON payloads); no `io.Seeker` body replay needed
- `VmConfig` and nested types mirror the CH REST API schema for type-safe configuration
- `Op` + `Code` dual classification distinguishes *what* failed from *which operation* failed, enabling precise error handling without string parsing

### Decision: VM Image Management (`pkg/image`)

The `pkg/image` package manages VM disk images, kernel parameters, and cloud-init configuration drives.

**Disk images:**
- Supports `raw`, `qcow2`, `vhdx`, `vmdk` formats
- Auto-detects format from file extension
- Validates path existence, format support, and optionally SHA-256 integrity

**Kernel command line:**
- Parses space-separated `key=value` and flag tokens
- Respects single and double quotes (values may contain spaces)
- Semantic validation: `root=` must be a valid block device path/UUID/LABEL, `ip=` must be a valid IP/CIDR
- Mutable API: `Get`, `Has`, `Set`, `AddFlag`

**Cloud-init:**
- Generates NoCloud datasource files (`user-data`, `meta-data`, `network-config`)
- Auto-generates meta-data from `InstanceID` and `Hostname` when not provided
- `ToImage()` returns a read-only `Image` descriptor for attaching to `vmm.DiskConfig`

**Design decisions:**
- Standard library only — no external image parsing or ISO generation libraries
- Format detection by extension rather than magic bytes (simpler, sufficient for operator-managed images)
- SHA-256 verification is optional but recommended for production images
- Cloud-init generates files on disk rather than an ISO (can be packaged into an ISO by the operator if desired)

### Decision: Network Interface Management (`pkg/network`)

The `pkg/network` package manages virtio-net interfaces for a single VM: creation, removal, QoS bandwidth limits, and an append-only audit log of every mutation.

**Key features:**
- **Interface lifecycle** — `AddInterface`, `RemoveInterface`, `Get`, `List`, `Count`. Auto-generates `ethXXXX` IDs and locally-administered unicast MACs when omitted
- **QoS bandwidth limits** — `SetQoS` applies per-interface ingress/egress rate limits in bytes/sec
- **Append-only audit log** — Every mutation is persisted as a JSON line with timestamp, operation, interface ID, details, and caller frame (`file:line func`). The log is created on `NewManager` and appended under a separate mutex to avoid blocking interface reads
- **Thread-safe concurrency** — `sync.RWMutex` guards the interface map; a separate `sync.Mutex` serialises audit log writes. ID generation uses `crypto/rand` with collision retry inside the write lock, guaranteeing unique IDs even under high concurrency
- **Deep-copy safety** — All getters return cloned `*Interface` values so callers cannot mutate internal state
- **Structured errors** — `*network.Error` carries `Op` (which operation failed) and `Message`, with `IsNetworkError` for type assertions

**Design decisions:**
- Standard library only — no external networking or MAC generation libraries
- `ethXXXX` IDs are 4 lowercase letters (456,976 namespace). Collision retry inside the lock makes this sufficient for typical VM workloads (< 1,000 interfaces)
- MAC addresses use `crypto/rand` with fallback to time+pid seed; first octet is masked to `02:xx:xx:xx:xx:xx` (locally administered, unicast)
- Audit log is line-delimited JSON (NDJSON) for easy streaming with `jq` or log shippers
- `runtime.Caller` captures the immediate caller of each public method for audit traceability without manual annotation

### Decision: Disk and Snapshot Management (`pkg/storage`)

The `pkg/storage` package manages virtio-blk disks for a single VM: hot-plug, hot-unplug, snapshot creation, and snapshot restore. Every operation is logged to a dedicated `storage-ops.log` with duration and outcome.

**Key features:**
- **Disk lifecycle (hot-plug / hot-unplug)** — `AddDisk` and `RemoveDisk` manage the attachment state. Auto-generates `diskXXXXXX` IDs when omitted. Disk paths are validated via the existing `pkg/image` package (format detection, existence, path resolution)
- **Snapshot creation** — `CreateSnapshot` performs a buffered file copy from the source disk to a destination path. Snapshot metadata (ID, size, timestamp) is recorded in the manager's snapshot map
- **Snapshot restore** — `RestoreSnapshot` copies a snapshot file back over the current disk path, overwriting it atomically via buffered copy
- **Operation log with duration** — Every operation is appended as NDJSON with `timestamp`, `operation`, `disk_id`, `duration_ms`, `outcome` (`success`/`error`), `message`, and `caller` frame. The dedicated `storage-ops.log` is separate from the application log so operators can grep storage events in isolation
- **Thread-safe concurrency** — `sync.RWMutex` guards both the disk and snapshot maps; a separate `sync.Mutex` serialises log writes. ID generation uses `crypto/rand` with collision retry inside the write lock
- **Deep-copy safety** — All getters return cloned `*Disk` and `*Snapshot` values so callers cannot mutate internal state
- **Structured errors** — `*storage.Error` carries `Op` and `Message`, with `IsStorageError` for type assertions

**Design decisions:**
- Reuses `pkg/image.Image.Validate()` for disk path/format validation rather than duplicating logic
- Snapshots are file copies (not COW or qcow2 internal snapshots) — this is the simplest portable approach for a management API and aligns with Cloud Hypervisor's raw/qcow2 backing file model
- `storage-ops.log` is separate from the general audit log so storage-specific SLOs and alerting can target one file
- `Duration` is rounded to microseconds and stored as a `time.Duration` (nanoseconds) so it is machine-parseable without string parsing
- Snapshot IDs use an 8-character alphanumeric namespace (2.8 trillion combinations) for uniqueness without UUID length

### Decision: Resource Hot-Resize Management (`pkg/resources`)

The `pkg/resources` package manages vCPU and memory configuration for a single VM: immediate in-memory updates, persistent state storage, limit and boundary validation, and detailed resize event logging.

**Key features:**
- **Hot-resize operations** — `ResizeCPU` and `ResizeMemory` validate the new value against configurable limits, update in-memory state immediately, persist atomically to a JSON state file, and append an operation log entry
- **Limit and boundary validation** — `Limits` struct defines min/max CPU count and memory size, plus memory alignment step (default 4 MB). `ValidateCPU` and `ValidateMemory` return structured errors describing which constraint was violated
- **Persistent state** — Current `Config` (CPUs, MemoryMB) is written atomically via temp-file-and-rename on every successful resize. On `NewManager`, existing state is loaded automatically; defaults (1 CPU, 256 MB) are used when no state file exists
- **Operation log with before/after/source** — Every resize event is appended as NDJSON with `timestamp`, `operation`, `source` (e.g. `"api"`, `"auto-scaler"`), `before`, `after`, `duration_ms`, `outcome`, `message`, and `caller` frame. The dedicated `resources-ops.log` allows operators to track capacity changes and attribute them to specific request sources
- **Rollback on persist failure** — If the atomic state write fails, the in-memory value is rolled back to the previous value before the error is returned, preventing config drift
- **Thread-safe concurrency** — `sync.RWMutex` guards config and limits; a separate `sync.Mutex` serialises log writes
- **Deep-copy safety** — `Config()` returns a cloned value so callers cannot mutate internal state
- **Structured errors** — `*resources.Error` carries `Op` and `Message`, with `IsResourcesError` for type assertions

**Design decisions:**
- State is a single JSON file (`resources.json`) rather than a database — this is sufficient for a single-VM manager and eliminates an external dependency
- Atomic state writes via temp-file + rename prevent corruption if the process crashes during a resize
- Memory alignment is configurable (default 4 MB) to match Cloud Hypervisor's page-size requirements without hard-coding assumptions
- Empty `source` defaults to `"unknown"` so logs are always attributable even when callers omit the origin
- `Duration` is rounded to microseconds and stored as a `time.Duration` (nanoseconds) for machine-parseability

### Decision: OpenAPI Documentation (`swaggo/swag`)

The API is documented with an OpenAPI 3.0 spec generated from inline Go annotations by [`swaggo/swag`](https://github.com/swaggo/swag).  This keeps the specification co-located with the code so that adding or changing an endpoint automatically updates the spec.

**How it works:**
1.  Handler methods carry `// @Summary`, `// @Tags`, `// @Param`, `// @Success`, `// @Failure`, `// @Router` comments above their signatures.
2.  `make swagger` (or `swag init -g cmd/ch-api/main.go`) parses the codebase and emits:
    - `docs/docs.go` — Go source that embeds the JSON spec
    - `docs/swagger.json` — raw OpenAPI JSON
    - `docs/swagger.yaml` — raw OpenAPI YAML
3.  `main.go` imports `_ "github.com/org/ch-api/docs"` so the embedded spec is registered at runtime.
4.  The Swagger UI is served at `GET /api/docs` via `http-swagger/v2`.

**CI enforcement:**
The GitHub Actions workflow runs `swag init` and then `git diff --exit-code docs/`.  If the committed spec diverges from the generated output, the build fails.  This guarantees that every PR either updates the annotations *and* re-generates the spec, or leaves both untouched.

### Decision: Event Log (`pkg/eventlog`)

The `pkg/eventlog` package emits human-readable changelog-style entries for every VM mutation.  Each entry captures the event type, VM ID, actor, and the before/after state of the VM.

**Outputs:**
- **Rotating log file** — `data/events-YYYY-MM-DD.log` rotates daily and contains one formatted line per entry.
- **Syslog** — entries are also sent to the local syslog daemon at `LOG_INFO | LOG_DAEMON` with the tag `ch-api`.

**Entry format:**
```
2026-05-13T21:30:00Z [boot] vm-abc by alice: status=created name="test-vm" cpus=1 mem=256MB disks=1 net=0 → status=running name="test-vm" cpus=1 mem=256MB disks=1 net=0
```

**Integration:**
- `Service` holds an `eventlog.Writer` (optional; nil disables emission).
- `CreateVM`, `DeleteVM`, and `UpdateVM` emit entries with the full VM state.
- Lifecycle transitions (`BootVM`, `PauseVM`, `ResumeVM`, `ShutdownVM`, `RebootVM`) emit entries with the before/after status.
- Errors do not block the operation; a failing syslog connection is silently ignored.

**Design decisions:**
- Human-readable text rather than structured JSON — operators can `tail -f` the file and understand changes immediately.
- Daily rotation aligns with the audit package's rotation strategy, making log archival simple.
- Dual output (file + syslog) provides local durability and centralised aggregation without extra infrastructure.
- The `eventlog.Writer` interface allows tests to pass `nil` so that event logging is completely opt-in.

## Dependency Policy

- **Deliberate dependencies only.** The project starts with the standard library plus `zerolog` for structured logging. Every external package is recorded in `go.mod` and justified in `ARCHITECTURE.md`.
- Prefer standard library solutions over third-party alternatives when practical.
- `zerolog` is chosen over `log/slog` for higher performance, richer event API, and mature ecosystem integrations (e.g. Loki, Splunk).
- `viper` is chosen for configuration management because it provides a battle-tested layered config system (defaults → file → env) with built-in hot-reload via `fsnotify`, saving us from writing and maintaining a custom config parser and file watcher.
